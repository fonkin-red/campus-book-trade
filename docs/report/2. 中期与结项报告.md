## 一、项目概述【中期必填】

### 1. 项目背景

随着高校教材更新换代频繁以及课外阅读需求增加，学生手中常常会积压大量闲置二手书籍。传统的毕业季摆摊、班级群或 QQ 群发布交易信息的方式存在信息分散、检索困难、交易状态不透明、联系效率较低等问题。

本项目旨在构建一个面向校园内部的二手图书交易平台，通过线上发布、浏览、收藏、购物车、下单、订单流转和后台管理等功能，盘活校内闲置图书资源，降低学生购书成本，促进绿色阅读与资源循环利用。

### 2. 项目目标

本系统以 Web 应用形式提供校园二手图书交易服务，核心目标如下：

1. 信息透明化：提供图书标题、作者、ISBN、分类、新旧程度、原价、售价、封面、详情描述、卖家信息等展示。
2. 交易流程化：支持买家浏览图书、收藏图书、加入购物车、生成订单、模拟付款、卖家发货、买家确认收货。
3. 管理便捷化：提供管理员后台，对用户状态、图书分类、订单状态、系统公告进行维护。
4. 协作规范化：使用 Git 进行版本控制，保留主要开发过程和问题修复记录。

### 4. 主要业务流程

用户登录流程：

1. 用户输入用户名和密码。
2. 前端调用 `/user/login` 接口。
3. 后端根据用户名查询用户，校验账号状态和 BCrypt 密码。
4. 校验通过后生成 JWT Token，并返回用户基础信息。
5. 前端保存 Token 和用户信息，后续请求自动携带认证信息。

图书发布流程：

1. 用户登录后进入发布图书页面。
2. 填写图书基础信息并上传封面图片。
3. 前端调用文件上传接口获得图片访问地址。
4. 前端提交图书信息到后端。
5. 后端写入 `book` 表，图书状态默认为在售。

订单交易流程：

1. 买家在图书详情页或购物车中发起下单。
2. 后端校验图书是否在售、是否为本人发布。
3. 生成订单快照，状态为待付款。
4. 买家模拟付款后，订单进入待发货状态，同时图书状态变更为已售出。
5. 卖家或管理员执行发货，订单进入已发货状态。
6. 买家确认收货后，订单进入已完成状态。
7. 未完成订单可取消；已付款或已发货订单取消时，图书恢复为在售。

## 三、系统设计【中期必填】

### 1. 系统架构

本系统采用前后端分离的 B/S 架构，由前端展示层、后端服务层和数据存储层组成。

**前端展示层**

- 基于 Vue 3 构建单页面应用，使用 Vite 作为开发和构建工具。
- 使用 Vue Router 管理页面路由，包括首页、登录、注册、详情、发布、购物车、订单、收藏、个人中心和管理员后台。
- 使用 Pinia 管理用户登录状态。
- 使用 Axios 封装 HTTP 请求，统一处理接口基础路径、Token 请求头和异常提示。
- 使用 Element Plus 搭建表单、表格、按钮、消息提示、弹窗等界面组件。

**后端服务层**

- 基于 Spring Boot 3.5.14 提供 RESTful API。
- 采用 Controller、Service、Mapper 分层结构。
- Controller 负责接收请求与返回统一响应。
- Service 负责业务规则校验和事务处理。
- Mapper 使用 MyBatis 注解方式访问 MySQL 数据库。
- 使用 LoginInterceptor 校验 JWT Token，并向请求对象注入 `userId`、`username`、`role`。
- 使用 BCrypt 进行密码加密，使用 JWT 实现无状态登录。

**数据存储层**

- 使用 MySQL 保存用户、图书、分类、购物车、订单、收藏和公告数据。
- 使用 `utf8mb4` 字符集，支持中文、特殊字符和扩展字符。
- 对用户名、订单号、用户-图书关系等字段建立唯一约束，避免重复数据。
- 对图书分类、卖家、状态、发布时间等字段建立索引，提高常用查询性能。

## 四、系统实现【中期部分填写；结项补全】

### 1. 关键技术

#### 1.1 JWT 无状态认证

系统登录成功后由后端生成 JWT Token，Token 中包含用户 ID、用户名和角色。前端将 Token 保存在本地，并在后续请求中通过 Axios 请求拦截器自动携带。后端 LoginInterceptor 对需要登录的接口进行统一校验，校验通过后将用户信息注入请求对象，业务层可直接获取当前用户身份。

该方式适合前后端分离项目，不需要在服务端维护 Session，接口扩展和前端调用都较方便。

#### 1.2 BCrypt 密码加密

用户注册时，后端不会保存明文密码，而是使用 BCryptPasswordEncoder 生成不可逆密文。登录时通过 `passwordEncoder.matches` 对用户输入密码和数据库密文进行校验。这样即使数据库泄露，也不会直接暴露用户原始密码。

#### 1.3 前端路由守卫

前端在路由中为需要登录的页面配置 `requiresAuth`，为管理员后台配置 `requiresAdmin`。每次路由跳转前检查本地 Token 和用户角色，不符合条件则提示用户并跳转到首页或登录页。

#### 1.4 MyBatis 注解式数据访问

后端 Mapper 层使用 MyBatis 注解编写 SQL，配合实体类字段自动映射，减少 XML 配置文件数量。常用查询包括图书列表、分类筛选、关键词搜索、订单列表、收藏列表等。

#### 1.5 文件上传与清理

图书封面图片通过文件上传接口保存到后端 `uploads` 目录，数据库只保存图片访问地址。删除图书时，后端会尝试删除关联封面和附加图片，减少无效文件堆积。

#### 3.1 用户登录认证（后端）

```java
public LoginResponse login(LoginRequest req) {
    User user = userMapper.selectByUsername(req.getUsername());
    if (user == null || user.getStatus() == 0) {
        throw new RuntimeException("用户不存在或已被禁用");
    }
    if (!passwordEncoder.matches(req.getPassword(), user.getPassword())) {
        throw new RuntimeException("密码错误");
    }
    String token = jwtUtil.generateToken(user.getId(), user.getUsername(), user.getRole());
    LoginResponse resp = new LoginResponse();
    resp.setToken(token);
    LoginResponse.UserInfo info = new LoginResponse.UserInfo();
    info.setId(user.getId());
    info.setUsername(user.getUsername());
    info.setNickname(user.getNickname());
    info.setPhone(user.getPhone());
    info.setEmail(user.getEmail());
    info.setAvatar(user.getAvatar());
    info.setRole(user.getRole());
    resp.setUser(info);
    return resp;
}
```

说明：该方法完成用户查询、账号状态判断、密码密文校验、JWT 生成和用户信息返回，是登录功能的核心逻辑。

```java
@Transactional
public void pay(Long id, Long buyerId) {
    OrderInfo order = orderMapper.selectById(id);
    if (order == null || !order.getBuyerId().equals(buyerId)) {
        throw new RuntimeException("无权操作");
    }
    if (order.getStatus() != 0) {
        throw new RuntimeException("订单状态不允许付款");
    }
    Book book = bookMapper.selectById(order.getBookId());
    if (book == null || book.getStatus() != 1) {
        throw new RuntimeException("图书不存在或已下架");
    }
    order.setStatus(1);
    order.setPaymentTime(LocalDateTime.now());
    orderMapper.updateStatus(order);
    book.setStatus(2);
    bookMapper.updateById(book);
}
```

说明：付款成功后订单状态变为“已付款待发货”，同时图书状态变为“已售出”，避免同一本书被重复购买。

### 3. 不足与改进方向

1. 交易支付仍为模拟流程，未接入真实支付平台。

5. 图片仍存储在本地目录，正式部署时可迁移到对象存储或静态资源服务。
6. 搜索目前以关键词模糊匹配为主，后续可优化为全文检索和多条件排序。
